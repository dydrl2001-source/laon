# 알렉스의 7원칙 — 거버넌스 차원의 불변원칙

> **출처**: [aldegad/alex-core-invariants](https://github.com/aldegad/alex-core-invariants) (영문 canonical / 한국어 derived)
> **적용 범위**: D 드라이브 모든 자동화 프로젝트 (manifest.yaml 등록 전체).
> **위반 검출**: 코드 리뷰 시 `/review` `/full-review` 에이전트가 항목별 검증.

## 왜 이 정책을 거버넌스로 박는가

7원칙은 *디버깅 잘하는 법*이 아니라 **"애초에 디버깅이 줄어들게 시스템을 짓는 법"**. 사후 처리가 아니라 사전 예방. 박광우님 자동화 14+개가 한 사람(원장님) 운영이므로, 각 프로젝트마다 매번 판단하는 대신 거버넌스 정책 한 곳에 박아 자동 상속 — 글로벌 `~/.claude/CLAUDE.md` → `D:\메인작업\CLAUDE.md` → 본 파일이 모든 세션에 자동 로드.

## 7원칙 (Required — 모든 프로젝트 의무)

### 1. SSoT — Single Source of Truth (단일 진실의 원천)

**한 줄**: 진실이 둘이면 어느 모델로 읽어도 둘이다.

**규칙**:
- 시스템이 의존하는 모든 진실은 단 하나의 canonical 소유자만 둔다
- 동시 존재하는 두 개의 canonical 경로는 금지
- 캐시·인덱스가 어긋나면 두 번째 진실을 만들지 말고, live truth로부터 canonical을 복구

**박광우님 자동화 적용 사례**:
- ✅ `manifest.yaml` = 14+개 프로젝트 인벤토리의 SSoT. `~/.claude/review_targets.json`은 derivative
- ✅ NotebookLM 평가매뉴얼 = 평가 기준의 SSoT (자체 지식 단독 결론 금지)
- ✅ 케어포 = 어르신 상태 데이터의 SSoT

**위반 신호**: "케어포에도 있고 별도 엑셀에도 있는데 값이 다름" 같은 상황.

### 2. SoC / SRP — Separation of Concerns / Single Responsibility (관심사·책임 분리)

**한 줄**: 혼재된 책임은 모든 리팩터링을 버티다가 결국 아무것도 못 바꾸는 지점에 도달한다.

**규칙**:
- 서로 다른 관심사는 분리된 경계에 둔다
- 한 경계가 두 관심사를 짊어지면 쪼갠다
- 슬래시커맨드: 한 커맨드 = 한 책임 (이미 `policies/슬래시-커맨드.md` 정책)
- 에이전트: 한 에이전트 = 한 도메인 (이미 `policies/에이전트-병렬.md` 정책)

**박광우님 자동화 적용 사례**:
- ✅ `/점검` / `/운영위원회 작성` / `/매일아침` / `/백업확인` — 각 명령 단일 책임
- ✅ 도메인 3 + 엔지니어급 5 에이전트 — 각자 한 도메인

**위반 신호**: 한 함수에서 데이터 수집 + 가공 + 보고서 작성 + 발송까지 다 하는 코드.

### 3. Consistency (일관성)

**한 줄**: 모순은 시간이 지나도 낫지 않는다. 누적될 뿐.

**규칙**:
- 데이터·상태·표현이 시스템 전반에서 서로 모순되면 안 됨
- 발견 즉시 정정 — 미루지 말 것

**박광우님 자동화 적용 사례**:
- ✅ 케어포 점검 5가지 동시 검증 중 **"체크 ↔ 서술 일치"** 항목이 정확히 이 원칙
- ✅ 블로그 표기 정책 ("푸른초장요양원2" 내부 vs "푸른초장요양원" 외부) — 경계별 일관성 명시

**위반 신호**: 1관 데이터에선 "박○○ 어르신", 2관에선 "박씨 어르신" 식으로 표기 다름.

### 4. Atomicity (원자성) — 🔧 보강 영역

**한 줄**: 반쪽짜리 상태는 아무도 선언하지 않은 두 번째 진실이다.

**규칙**:
- 변경은 완전히 끝나거나 깔끔하게 롤백되어야 한다
- 반쪽짜리 상태를 외부에 노출 금지
- 다단계 작업이 중간에 끊기면 그때까지 만든 부산물도 정리

**박광우님 자동화 보강이 필요한 곳**:
- 🟡 백업: ZIP 9개 모두 완성 후 sentinel 생성 — 부분 적용됨. 그러나 robocopy 미러링이 중간에 끊기면 일부 파일만 옮겨진 상태로 남을 수 있음 → **다음 자동화 합류 시 sentinel 보장 패턴 일반화 권고**
- 🔴 블로그자동화: 글 게시 성공했는데 사진 업로드 실패 시 반쪽 게시물 남는지 확인 필요
- 🔴 케어포작성: 8-X 항목 자동 작성 중간 실패 시 일부만 입력된 상태 검토 필요

**적용 패턴**:
- 작업 시작 시 `processing` 마커 생성 → 완료 시 `done` 마커로 교체 → 다음 실행은 `done` 없으면 정리부터
- 또는 임시 경로에 작업 후 마지막에 원자적 rename

### 5. Idempotency (멱등성)

**한 줄**: 멱등성이 없으면 재시도는 복구가 아니라 손상이 된다.

**규칙**:
- 같은 요청을 반복해도 결과는 같은 값으로 수렴
- 재실행 시 부작용(중복 발송·중복 저장) 없음

**박광우님 자동화 적용 사례**:
- ✅ `/매일아침` 두 번 쳐도 그날 결과 동일 (Gmail 초안만 2개 생기는 정도 — 메인 결과는 멱등)
- ✅ `/백업확인` 여러 번 실행해도 sentinel 한 개

**위반 신호**: `/매일아침` 두 번 치면 어르신 위험 알림이 두 번 발송됨 같은 패턴.

**박광우님 자동화 보강이 필요한 곳**:
- 🟡 카톡보고 자동화 재실행 시 중복 보고 검증 필요
- 🟡 희망이음반자동 재실행 시 중복 입사보고 검증 필요

### 6. No Silent Fallback (조용한 폴백 금지)

**한 줄**: 조용한 폴백이 발생하는 순간 코어는 더 이상 코어가 아니다.

**규칙**:
- `fallback`, `legacy`, `shadow path`가 실패를 조용히 숨기면 금지
- 가용성을 위한 명시적 failover는 허용 — 단 **관측 가능**해야 하고 canonical 진실을 바꾸지 않아야 함
- "그냥 안 보임"이 아니라 "어떤 이유로 실패함" 명시

**박광우님 자동화 적용 사례**:
- ✅ `/매일아침`에서 NotebookLM 끊겼을 때 → 가짜 마감 만들지 않고 **"평가 마감 — NotebookLM 미연결, 매뉴얼 확인 불가"** 명시 (정확히 이 원칙 준수)
- ✅ `notify.py` 가 winotify 미설치 시 조용히 스킵 — 단 호출 측 작업은 안 막음 (이건 알림 영역의 의도된 failover, canonical 산출물엔 영향 없음 → 허용 범위)

**위반 신호**: 평가매뉴얼 조회 실패 시 자체 지식으로 답을 만들어내는 패턴.

### 7. Doc-first & Plan-first (문서·계획 먼저) — 🔧 새 영역 확장

**한 줄**: 추측에 쓰는 한 시간은 안 읽기로 한 1분이다. 다시 짓는 데 쓰는 한 주는 plan에 쓰지 않은 30분이다.

**규칙 (a) Doc-first**:
- 표준화된 도메인 문제는 **공식 문서 검색이 첫 행동**
- 가설 우선이 정당화되는 건 내부 코드 / 내부 도메인 룰 / 문서 미명시 edge 일 때만

**규칙 (b) Plan-first**:
- 비-trivial 작업은 첫 commit 전 plan으로 적는다
- plan의 phase가 doc-first를 적절한 진입점에서 트리거 + 깨졌을 때 복구 sequence 제공 + ad-hoc fix→commit→deploy 사이클을 차단
- trivial한 1-step 작업만 plan 생략 가능

**박광우님 자동화 적용 사례**:
- ✅ 케어포·평가매뉴얼·법령 = NotebookLM 우선 (`policies/케어포-도메인.md` Fallback 절차까지 명문화)
- ✅ Opus 4.7 활용 7원칙 중 #3 "복잡한 작업은 Plan 먼저" (이미 정책)

**박광우님 자동화 보강이 필요한 곳**:
- 🟡 **새 영역**(처음 쓰는 도구·새로 시도하는 자동화)에서도 Doc-first 의식적 적용. 익숙한 케어포·평가매뉴얼은 잘 되어 있으나, 신규 영역(예: 새 MCP 서버, 새 라이브러리)에서 추측 우선 패턴 경계
- 🟡 다단계 작업은 EnterPlanMode 적극 활용 (메인 어시스턴트에 plan 강제 패턴 권고)

## 모든 프로젝트 의무화 명령

본 정책은 글로벌 import 체인으로 자동 상속:

```
~/.claude/CLAUDE.md
  └ @D:\메인작업\CLAUDE.md
      └ @policies/알렉스-7원칙.md  ← 본 파일
```

즉 어느 폴더에서 Claude Code를 켜든 본 7원칙이 시스템 컨텍스트에 포함됨. 각 프로젝트가 별도 import할 필요 없음.

**위반 검출 책임**:
- `/review` (5에이전트 일상 리뷰) — 각 원칙별 코드 스캔
- `/full-review` (13에이전트 분기 리뷰) — 8종 엔지니어급 에이전트가 항목별 심층 검증
- `엔지니어-스태프-SWE` 에이전트 — SoC/SRP/Consistency 위반 우선 식별
- `엔지니어-AI프롬프트` 에이전트 — Doc-first/Plan-first 위반 식별
